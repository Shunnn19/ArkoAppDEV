AUTHENTICATION ALGORITHM DOCUMENTATION
========================================

1. PRIMARY ALGORITHM: bcrypt (Password Hashing)
-------------------------------------------------

Algorithm:   bcrypt with Blowfish cipher (Eksblowfish)
PHP Constant: PASSWORD_BCRYPT
Cost Factor:  10 (default; 2^10 = 1024 Blowfish key expansion rounds)
Output:      "$2y$10$" + 22-char base64 salt + 31-char base64 hash = 60 characters total

Description:
  bcrypt is a password-hashing function designed by Niels Provos and David
  Mazières in 1999. It is based on the Blowfish cipher's key schedule and
  incorporates a salt to protect against rainbow table attacks, and an
  adaptive cost factor that increases computational cost to thwart
  brute-force attacks.

Purpose:
  To securely hash user passwords before storing them in the database so
  that plaintext passwords are never persisted. When a user logs in, the
  provided password is verified against the stored bcrypt hash using
  PHP's password_verify() function, which performs constant-time
  comparison to prevent timing attacks.

How strong is it:
  - bcrypt at cost=10 requires ~70-100ms per hash on modern hardware.
  - A brute-force attempt against a bcrypt hash is significantly slower
    than against MD5 or SHA-1 (by a factor of 10^5 or more).
  - GPU/ASIC resistance is high because bcrypt requires ~4 KB of memory
    per computation, making parallel implementations less efficient.
  - NIST and OWASP both recommend bcrypt as a safe choice for password
    storage (along with Argon2 and scrypt).

Why it was chosen:
  - Built into PHP 5.5+ (password_hash() / password_verify()), requiring
    zero external dependencies.
  - Industry standard for web application password storage.
  - Adaptive cost allows future-proofing by increasing the cost factor
    as hardware improves.
  - More than adequate for a museum operations system — the threat model
    is credential theft via database breach, not nation-state attacks.

Formula (how it works):

  1. Salt Generation:
     A cryptographically secure pseudo-random 16-byte (128-bit) salt is
     generated using /dev/urandom or equivalent OS entropy source.

  2. Key Setup (Eksblowfish):
     The salt and password are fed into the Blowfish key schedule, which
     performs 2^cost (1024 at cost=10) iterations of the Blowfish cipher
     to produce an expanded key array.

  3. Hashing:
     The expanded key is used to encrypt the string "OrpheanBeholderScryDoubt"
     64 times in ECB mode using Blowfish.

  4. Output Format:
     "$2y$10$[22-char salt][31-char hash]"  (60 characters total)
     - "$2y$" = bcrypt version identifier
     - "10"   = cost factor (log2 of iterations)
     - Next 22 chars = base64-encoded salt
     - Final 31 chars = base64-encoded hash

  5. Verification (login):
     password_verify($input_password, $stored_hash) extracts the salt and
     cost from the stored hash, recomputes the hash with the input password,
     and does a constant-time comparison against the stored hash.


2. SECONDARY ALGORITHM: 256-bit CSPRNG Token (Session Authentication)
-----------------------------------------------------------------------

Algorithm:   Cryptographically Secure Pseudo-Random Number Generator
PHP Function: random_bytes(32) → 32 bytes → bin2hex() → 64-char hex string
Token Size:  256 bits (32 bytes) → 64 hexadecimal characters
Expiry:      24 hours (default) or 7 days (with "rememberMe" flag)

Description:
  Instead of JSON Web Tokens (JWT) or session cookies, this system uses a
  simple 256-bit random token generated by PHP's random_bytes() function,
  which draws entropy from the operating system's CSPRNG (e.g., /dev/urandom
  on Linux, CryptGenRandom on Windows).

Purpose:
  To authenticate API requests after login. Upon successful login, the
  server generates a random token, stores it in the users.auth_token column
  with an expiry timestamp, and returns it to the client. The client sends
  this token as an "Authorization: Bearer <token>" header on subsequent
  requests. The server looks up the token in the database to identify the
  user.

How strong is it:
  - 256-bit random token = 2^256 possible values, effectively immune to
    brute-force guessing.
  - PHP's random_bytes() uses the OS kernel's CSPRNG, which is the same
    source used for TLS/SSL key generation.
  - Tokens are stored as VARCHAR(128) in the database; the 64-char hex
    representation is safe for URLs, headers, and database storage.
  - Tokens have a finite expiry (24h / 7d) and are invalidated on logout
    (set to NULL in the database).

Why it was chosen over JWT:
  - Simpler implementation: no signing keys, no JWT library, no token
    refresh logic.
  - Fully revocable: deleting/expiring the token in the DB immediately
    invalidates the session (JWT is stateless and cannot be revoked
    without a blocklist).
  - Server-controlled: the server is the single source of truth for
    session validity.
  - Appropriate for this application: a single-server PHP + MySQL setup
    where database lookups are negligible overhead.

Formula (how it works):

  1. Token Generation (login):
     $token = bin2hex(random_bytes(32));
     // Produces a 64-character hex string like:
     // "a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0c1d2e3f4a5b6c7d8e9f0a1b2"

  2. Token Storage:
     UPDATE users SET auth_token = ?, auth_token_expires = ? WHERE id = ?
     // Expiry is computed as NOW() + 24 hours (or + 7 days for rememberMe)

  3. Token Verification (on each protected API request):
     SELECT id, name, email, role FROM users
     WHERE auth_token = ? AND auth_token_expires > NOW()
     // If no row returned, token is invalid or expired → 401 response

  4. Token Invalidation (logout):
     UPDATE users SET auth_token = NULL, auth_token_expires = NULL WHERE id = ?
     // Sets token to NULL, making further requests with that token fail


3. DATA FLOW SUMMARY
---------------------

  Registration:
    User submits email + password → server bcrypt-hashes (cost=10) →
    stores hash in users.password_hash → user created

  Login:
    User submits email + password → server looks up user by email →
    password_verify() checks bcrypt hash → on success, generates
    256-bit token → stores token + expiry in users table →
    returns token + user info to client

  Authenticated Request:
    Client sends Authorization: Bearer <token> header →
    server calls getAuthUser() → SELECT ... WHERE auth_token = ?
    AND auth_token_expires > NOW() → if valid, returns user array

  Logout:
    Client calls POST /api/auth/logout → server calls requireAuth()
    to verify token → sets auth_token = NULL in database
