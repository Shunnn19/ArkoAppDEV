# Glossary of Technical Terms & Terminologies

| Term | Definition | Context |
|------|------------|---------|
| **Attendance Log** | A record of a visitor's arrival and departure at a museum, stored in the `ATTENDANCE_LOG` table with arrival time, departure time, entry type, and status. | Used by Staff to track who is on-site and by Curators to monitor occupancy. |
| **Auth Token** | A 256-bit cryptographically secure random string (`bin2hex(random_bytes(32))`) stored in `users.auth_token` that authenticates API requests via the `Authorization: Bearer` header. | Replaces session cookies. Stateless on the client, revocable server-side by deleting the token from the database. |
| **bcrypt** | A password-hashing function (Blowfish-based, Eksblowfish) with configurable cost factor. Used via PHP's `PASSWORD_BCRYPT` with cost=10. | NIST-recommended, GPU/ASIC-resistant. Produces a 60-character `$2y$10$...` hash. Every test user's password is hashed this way in seed.sql. |
| **Booking Status** | An ENUM on `VISITOR_BOOKING.bookingStatus` with values: `Pending`, `Confirmed`, `Cancelled`, `Modified`. Determines what actions are available (e.g., Log Arrival only for Confirmed). | Pending bookings await curator approval. Confirmed bookings can be modified (which resets them to Pending). Cancelled bookings are inactive. |
| **CORS (Cross-Origin Resource Sharing)** | HTTP headers set exclusively in `.htaccess` to allow the frontend (localhost:5173) to call the backend (localhost/museum-api). | Setting them in both `.htaccess` and PHP files causes duplicate-header errors in browsers. Only `.htaccess` sets them. |
| **CSPRNG** | Cryptographically Secure Pseudo-Random Number Generator. PHP's `random_bytes(32)` produces 256 bits of unbiased random data. | Used for auth token generation. The 64-character hex string has 2^256 possible values — computationally infeasible to brute-force. |
| **Data Dictionary** | The authoritative specification of all 7 entities (VISITOR, VISIT_SCHEDULE, VISITOR_BOOKING, STAFF_ASSIGNMENT, WALK_IN_VISITOR, ATTENDANCE_LOG, NOTIFICATION_RECORD) plus the `users` table, including column names, data types, ENUM values, and FK relationships. | Two intentional deviations: `timeSlot` is VARCHAR(20) not TIME (to store range format), and `userId` is VARCHAR(15) not VARCHAR(9) (to match external system IDs). |
| **Departure Time** | A nullable TIME column on `ATTENDANCE_LOG.departureTime`. NULL means the visitor is still on-site. When set, the entry is marked complete and on-site occupancy decreases. | Walk-in entries and pre-booked arrivals both track departure. The dashboard's "On-Site Now" count excludes entries with a departure time. |
| **Entry Type** | An ENUM on `ATTENDANCE_LOG.entryType` with values `Pre-booked` and `Walk-in`. Determines whether the log links to a `VISITOR_BOOKING` or a `WALK_IN_VISITOR` record. | Walk-in entries cannot be deleted from the logbook. Pre-booked entries are tied to a specific booking reference. |
| **En-dash** | The `–` character (U+2013) used in time slot range format (`09:00–10:00`). Stored as part of the VARCHAR(20) `timeSlot` field. | Must be supported via `SET NAMES utf8mb4` on the PDO connection, otherwise it gets corrupted during storage or retrieval. |
| **Hybrid Route** | An API route where POST is public but GET/PUT/DELETE require authentication. Used for `visitor-bookings` — anyone can submit a booking, but only authenticated users can view or modify them. | Controlled by the `$hybridRoutes` array in `backend/index.php`. |
| **Museum Context** | The currently selected museum that scopes all operations — schedules, bookings, staff assignments, attendance logs, and reports. Set via a dropdown in the sidebar. | All API queries filter by `museumSelection`. Changing context refreshes all data for the selected museum. |
| **My Booking** | A frontend tab that shows only the current session's booking references, stored in `localStorage['myBookingRefs']`. Does NOT allow searching by reference number or email to look up other people's bookings. | A privacy-by-design decision. Each session tracks its own bookings via the reference numbers returned during booking submission. |
| **PDO (PHP Data Objects)** | PHP's built-in database access layer. Used with prepared statements (`?` placeholders) to prevent SQL injection. | Every SQL query in the backend uses PDO prepared statements. No `mysqli_query`, no string interpolation. |
| **Prepared Statement** | A SQL query with `?` placeholders where values are bound separately, preventing malicious input from being interpreted as SQL code. | Example: `$stmt = $db->prepare('SELECT * FROM users WHERE email = ?'); $stmt->execute([$email]);` |
| **Public Route** | An API route that requires no authentication. Used for museums list, holidays, auth endpoints, visitors lookup, and visit schedules. | Defined in the `$publicRoutes` array in `backend/index.php`. Anyone can browse schedules without logging in. |
| **Role-Based Access Control (RBAC)** | Access restriction based on the `users.role` column (curator, staff, general, researcher, educator). The frontend routes users to role-specific dashboards and the backend enforces access at the API level. | App.tsx checks `user.role` to render the correct dashboard. The sidebar only shows navigation links appropriate to the role. |
| **Schedule Override** | Changing a `VISIT_SCHEDULE.status` from `Open` to `Closed`, `Holiday`, or `Blocked` (with a required reason). Affects real-time availability shown to visitors. | Curators use this to block slots for maintenance, holidays, or special events. The frontend shows the new status and reason in the scheduling interface. |
| **Seed Data** | Pre-populated test data in `database/seed.sql` that creates realistic records for all 7 tables plus 5 user accounts with known passwords. | Run after schema.sql to get a working system immediately. Includes bcrypt password hashes for all test accounts. |
| **Time Slot** | A 1-hour visit window stored as a VARCHAR(20) range string (`09:00–10:00`). Not a TIME type because it represents a range, not a point in time. | There are 7 slots per day: 09:00–10:00 through 16:00–17:00 (no 08:00–09:00). Defined in the frontend as `DEFAULT_TIME_SLOTS`. |
| **Visitor ID** | Auto-generated ID format `VIS-YYYY-NNN` for the `VISITOR` table. Created when a new visitor submits a booking or registers. | Separate from `users.id`. Visitors are not the same as user accounts — a visitor exists only as a booking contact. |
| **Walk-in** | A visitor who arrives without a pre-booked appointment. Recorded in the `WALK_IN_VISITOR` table and logged in `ATTENDANCE_LOG` with `entryType = 'Walk-in'`. | The staff interface auto-creates a `VISIT_SCHEDULE` if one doesn't exist for the selected time slot. Walk-ins cannot be deleted from the logbook. |
| **256-bit Token** | A 64-character hexadecimal string (256 bits of entropy) generated by `bin2hex(random_bytes(32))`. Used for API authentication. | 2^256 possible values. Stored in `users.auth_token` with an expiry timestamp in `users.auth_token_expires`. |
